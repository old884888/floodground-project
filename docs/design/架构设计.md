## 一、世界空间系统

### 1.1 核心概念

不是三个独立地图。是**同一套世界坐标，两个镜头焦距**。

```
┌──────────────────────────────────────────┐
│           L1 · 世界视图                   │
│   整个星球平面，emoji + 文字标注          │
│   势力在哪、你在哪、地形大概啥样          │
│   → 纯信息面板，后台批量更新全局事件      │
│   → 不操作、不走路、不战斗               │
└─────────────┬────────────────────────────┘
              │   同一坐标空间
              │   只是画面裁切 + 缩放不同
              │   没有任何"切场景"的概念
┌─────────────▼────────────────────────────┐
│           L2 · 区域视图                   │
│   现实气泡内：瓦片视野（默认 ~60×40 格）  │
│   走路、造房子、方向键怼人、营地管理      │
│   → 一切日常、一切操作都在这里           │
│   → 气泡内全 AI 模拟                     │
└──────────────────────────────────────────┘
```

核心信条一句话：**角色站在世界坐标上，世界在后台活着，镜头只是你去哪看的问题。**

### 1.2 世界坐标系统

- 整个星球 = 一张 2D 瓦片平面网格
- 每个瓦片有坐标 `(x: i64, y: i64)`，大但有限（默认 8000×8000 瓦片，足够走一辈子）
- 瓦片 = 地形 + 建筑 + 物品 + 角色 + 液体
- 世界切分为 **区块（Chunk）**：每个区块 16×16 瓦片，是生成/加载/序列化的最小单位
- 种子随机生成（每局不同），走 CDDA 那套 Perlin 噪声 + 生态区域

```
世界层级结构：

瓦片(Tile)  →  区块(Chunk, 16×16)  →  区域(Region, 8×8区块)  →  生态区(Biome)
(1格)           (生成/加载单元)          (128×128瓦片)              (512×512瓦片)

区块在内存中只有气泡附近的才驻留，剩下的序列化到磁盘。
```

### 1.3 现实气泡（Reality Bubble）

抄 CDDA，不废话：

| 层级 | 范围 | 模拟精度 | 干什么 |
|------|------|---------|--------|
| 气泡内 | 当前角色周围 60 格半径 | **全 AI**：寻路、需求、社交、战斗、工作、事件生成 | 一切活着的都在算 |
| 气泡外 | 60 格半径之外 | **批量结算**：按小时/天批量更新、不寻路、不发即时事件、角色冻结 | 势力移动、作物生长、尸体腐烂 |

规则：
- 气泡总是以**当前被控制的角色**为中心
- 切换控制对象 → 气泡平移，新中心周围全 AI 激活
- 区块离开气泡 → 序列化写入内存缓存 → 释放 CPU 和缓存行
- 区块进入气泡 → 从缓存反序列化 → 全 AI 激活
- 切换 L1 世界视图时气泡不消失（玩家只是在看全图，气泡里的人还在动）

### 1.4 营地系统

玩家在冒险模式中，在任意世界坐标划定一块区域作为营地：

```
Camp Mode 激活 → 在世界地图上拖矩形边界 → 边界内 = 营地

营地内的特殊规则：
├── 可以放置建筑（营地外不能随意建设）
├── 电力网络只存在于营地边界内
├── 作业队列管理器（Job Manager）自动分配任务
├── 所有归属殖民者在此区域内活动
├── 储物空间、农田、牢笼统一管理
└── 营地边界可扩展（消耗资源 + 大量时间）

营地不是"切走"的独立场景——
营地就是世界坐标上的一块矩形区域。
敌人能走到你营地门口撞墙。
建筑是实体瓦片，能挡住、能烧毁、能被围攻。
```

### 1.5 地形与世界生成

世界生成流程图：

```
世界种子(u64)
    │
    ▼
Perlin 噪声生成器
    │
    ├→ 海拔图 → 海洋/平原/山地
    ├→ 温度图 → 热带/温带/寒带
    ├→ 湿度图 → 森林/草原/荒漠/沼泽
    └→ 生态区分配 → 分配种族起始区域 + 造物者遗迹坐标
    │
    ▼
生成区块（Chunk 级别）
    │
    ├→ 基础地形瓦片（草地/沙地/水域/山壁）
    ├→ 资源分布（树木/矿物/草药）
    ├→ 野生动物（按生态区生成不同动物）
    ├→ 造物者遗迹（按层级概率放置）
    └→ 势力定居点（至少 1 个势力定居点/生态区）
```

生态区类型（来自世界观.md）：

| 生态区 | 地形倾向 | 资源倾向 | 势力偏好 |
|--------|---------|---------|---------|
| 平原 | 开阔草地 | 农作物、树木 | 人类 |
| 森林 | 密集树木 | 木材、野生动物 | 精灵 |
| 山地 | 岩石、矿脉 | 矿物 | 矮人 |
| 沼泽 | 水域、毒雾 | 草药、稀有植物 | 哥布林 |
| 荒漠 | 沙地、干涸 | 矿物、遗迹 | 龙裔 |
| 亡灵废土 | 灰烬、辐射 | 亡灵资源、禁忌科技 | 不死者 |

## 二、游戏模式

两个模式共享同一世界，按 Tab 或按钮切换。

```
┌─────────────────────────────────────────────────┐
│               Adventure Mode（默认）              │
│                                                  │
│  控制一个主角                                    │
│  WASD / 方向键 走路                               │
│  方向键攻击相邻瓦片敌人                           │
│  捡东西、开门、潜行、探索                         │
│  UI：角色面板 + 血条 + 附近瓦片 + 事件日志        │
│  → 按下 Tab 进入 Camp Mode                       │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│               Camp Mode                           │
│                                                  │
│  鸟瞰式殖民地管理                                 │
│  放建筑、划农田、指派作业                         │
│  控制其他殖民者（需忠诚度达标）                   │
│  作业队列自动运转                                 │
│  UI：①殖民地面板 ②建筑列表 ③殖民者列表           │
│       ④作业队列 ⑤资源统计                        │
│  → 再按 Tab 或直接点一个角色切回 Adventure Mode   │
└─────────────────────────────────────────────────┘
```

模式影响：
- 输入映射不同（Adventure = 角色移动 + 方向攻击, Camp = 鼠标选择 + 指令）
- UI 面板不同（Adventure 看角色, Camp 看全局）
- Tick 推进规则相同（Camp Mode 下时间照样走，敌人照样来）
- 现实气泡规则相同（始终追着当前控制的主角）

## 三、ECS 架构

### 3.1 总体架构图

```
┌──────────────────────────────────────────────┐
│              UI (egui, 双模式)                 │
│   L2 区域视图 + L1 世界视图 + Camp 管理面板    │
└──────────────────┬───────────────────────────┘
                   │ 事件订阅 + 只读查询
┌──────────────────▼───────────────────────────┐
│            Event Bus (事件总线)                │
│  enum GameEvent { Tick, BuildPlaced,         │
│    EnemyAttacked, CampBoundaryChanged, ... }  │
│  所有系统只通过 EventBus 通信                  │
│  禁止直接读写另一个 System 的 Component        │
└────┬──────────┬──────────┬──────────┬────────┘
     │          │          │          │
┌────▼───┐ ┌───▼────┐ ┌───▼────┐ ┌───▼────────┐
│WorldSys│ │CharSys │ │CampSys │ │ CombatSys  │
│(区块   │ │(需求   │ │(营地   │ │ (方向攻击  │
│ 生成)  │ │  AI)   │ │ 管理)  │ │  伤害)     │
└───┬────┘ └───┬────┘ └───┬────┘ └───┬────────┘
    │          │          │          │
┌───▼──────────▼──────────▼──────────▼──────────┐
│               ECS World                        │
│  Entity: 角色/建筑/物品/尸体/瓦片               │
│  Component: HP, Mood, Will, Position, Tile,    │
│             Building, Corpse, Job, Memory, ...  │
│  Resource: WorldMap, CampData, JobManager, ...  │
└────────────────────────────────────────────────┘
```

### 3.2 Entity 类型（用 Component Marker 区分）

| Entity | Marker Component | 说明 |
|--------|-----------------|------|
| 主角 | `Player` | 当前被控制的角色 |
| 殖民者 | `Colonist` | 归属营地的角色 |
| 俘虏 | `Captive` | 抓来的，有 Will 值 |
| 敌人 | `Hostile` | 不归你管的攻击性角色 |
| 瓦片 | `Tile` | 世界地表，有地形类型 |
| 建筑 | `Building` | 所有建筑实体，占瓦片 |
| 物品 | `Item` | 可拾取/交易的落在地上对象 |
| 尸体 | `Corpse` | 尸体，带腐烂计时器 |
| 区块 | `Chunk` | 世界生成/加载单元 |

### 3.3 Component 粒度（越小越好）

```
好的拆分：
    struct Position { x: i64, y: i64 }
    struct Health { hp: f32, max_hp: f32 }
    struct Mood { value: f32 }
    struct Hunger { value: f32 }
    struct Memory { entries: Vec<MemoryEntry> }
    struct TileType { terrain: TerrainKind }
    struct BuildingMarker { building_id: String }
    struct Loyalty { value: f32 }          // 忠诚度，>阈值才能控
    struct JobClaim { job_id: Uuid }

坏的拆分（上帝组件，滚）：
    struct Character {
        hp: f32, mood: f32, hunger: f32,
        skills: Vec<Skill>, memories: Vec<Memory>,
        equipment: Vec<Item>,
        // ... 50 个字段，缓存行爆炸，System 并行失效
    }
```

原因：ECS 的 System 会查询 "所有有 Health 的 Entity" 批量处理。组件越小，遍历时内存越紧凑，CPU 缓存命中率越高。bevy_ecs 内部就是 SoA，别自己造轮子。

### 3.4 关键 Component 定义

```rust
// —— 空间 ——
struct Position { x: i64, y: i64 }              // 世界坐标（所有实体都有）
struct Tile { terrain: TerrainKind }             // 瓦片地表类型
struct ChunkCoord { cx: i32, cy: i32 }           // 区块坐标（区块 Entity 用）

// —— 角色 ——
struct Player;                                   // Marker: 当前主角
struct Colonist { loyalty: f32, job_skill: ... } // 殖民者数据
struct Captive { will: f32, captivity_days: u32 }// 俘虏数据
struct Hostile;                                  // Marker: 敌人

// —— 营地 ——
struct CampBoundary { min_x: i64, min_y: i64, max_x: i64, max_y: i64 }
struct OwnedByCamp;
struct JobClaim { job_id: Uuid, status: JobStatus }
```

## 四、Event Bus 设计

所有系统之间通过事件通信。**任何系统不直接读写另一个 System 的 Component**，要改就发事件。

```rust
enum GameEvent {
    // —— 时间 ——
    Tick(u64),
    DayElapsed(u64),

    // —— 空间（新增） ——
    ChunkGenerated(ChunkCoord),
    ChunkLoaded(ChunkCoord),
    ChunkUnloaded(ChunkCoord),
    BubbleMoved { center: (i64, i64), radius: u32 },

    // —— 人物 ——
    CharacterSpawned { entity: Entity, pos: (i64, i64) },
    CharacterDied { entity: Entity, cause: CauseOfDeath, pos: (i64, i64) },
    CharacterMoved { entity: Entity, from: (i64, i64), to: (i64, i64) },
    MoodChanged { entity: Entity, delta: f32 },
    BreakdownTriggered { entity: Entity, breakdown_type: BreakdownType },

    // —— 游戏模式（新增） ——
    ModeSwitched { from: GameMode, to: GameMode },
    CampBoundaryPlaced { min: (i64, i64), max: (i64, i64) },
    CampBoundaryExpanded { new_min: (i64, i64), new_max: (i64, i64) },

    // —— 战斗 ——
    ActorAttacked { attacker: Entity, target: Entity, direction: Direction },
    DamageDealt { victim: Entity, amount: f32, body_part: BodyPart },
    EnemyDown { entity: Entity, pos: (i64, i64) },

    // —— 建造 ——
    BuildOrdered { building_id: String, pos: (i64, i64) },
    BuildProgressed { entity: Entity, progress: f32 },
    BuildCompleted { entity: Entity },

    // —— 作业队列 ——
    JobCreated { job_id: Uuid, job_type: JobType },
    JobClaimed { job_id: Uuid, worker: Entity },
    JobCompleted { job_id: Uuid, worker: Entity },

    // —— 黑暗 ——
    TortureStarted { actor: Entity, victim: Entity },
    TortureCompleted { victim: Entity, result: TortureResult },
    OrganHarvested { donor: Entity, organ_type: OrganType },
    CaptiveConverted { entity: Entity },

    // —— 世界 ——
    VisitorArrived { faction_id: String, pos: (i64, i64) },
    ReputationChanged { delta: i32 },

    // —— 系统 ——
    SaveRequested,
    LoadRequested(String),
}
```

规则：
- 事件是结构化的（不是字符串），编译时检查类型
- 事件可以序列化（为了存档/回放）
- EventBus 本身是一个 bevy Resource，System 通过 `EventWriter<T>` / `EventReader<T>` 读写

## 五、Tick 循环

游戏不是 real-time 的，是回合制 Tick 推进：

```
loop {
    handle_input();              // 按键/鼠标 → 当前模式的事件
    process_events();            // 处理积压的所有事件
    run_systems(tick_id);        // 所有 System 按序执行
    update_reality_bubble();     // 气泡位置检测，激活/冻结区块
    render_ui();                 // egui 渲染（按当前模式画不同面板）

    if player_clicked_play {
        advance_ticks(1);
    }
    if player_clicked_fast_forward {
        advance_ticks(100);     // 快进时 UI 只在首尾各刷一次
    }
}
```

System 执行顺序（注册在 schedule 里）：

```
01. WorldGenSystem          // 生成缺失区块（惰性生成）
02. BubbleSystem            // 气泡激活/冻结区块
03. InputSystem             // 解析输入 → 发事件
04. MovementSystem          // 角色移动 → 更新 Position
05. NeedsSystem             // 饥饿/口渴/疲劳更新
06. NatureSystem            // 作物生长、尸体腐烂、火扩散
07. JobManagerSystem        // 作业分配 + 进度更新
08. CombatSystem            // 伤害结算、死亡判定
09. DarkSystem             // 刑讯/手术/灵魂提取
10. FactionSystem           // 势力AI、巡逻移动、外交
11. NarrativeSystem          // 事件日志生成
12. ChunkSaveSystem          // 离开气泡的区块序列化
```

## 六、Job Manager（作业队列系统）

抄矮人要塞 + RimWorld，不抄全但抄精髓：

### 6.1 作业来源

```
作业产生：
├── BuildJob       ← 玩家在营地模式下放了建筑
├── HarvestJob     ← 作物成熟、树木砍伐标记
├── CraftJob       ← 玩家在工坊下单制造
├── HaulJob        ← 物品散落在地上（搬到储物区）
├── GuardJob       ← 治安值低、或指定瞭望塔警卫
├── ResearchJob    ← 研究台有空闲研究者
├── FeedJob        ← 俘虏需要喂食
├── BuryJob        ← 尸体需要处理
├── RepairJob      ← 建筑受损需要修
└── DarkJob        ← 刑讯/手术/灵魂提取排队
```

### 6.2 作业结构

```rust
struct Job {
    id: Uuid,
    job_type: JobType,
    priority: u8,                    // 0-10，越高越急
    pos: (i64, i64),                 // 执行位置（世界坐标）
    required_skill: Option<SkillKind>,
    required_tool: Option<String>,
    progress: f32,                   // 0.0 - 1.0
    claimed_by: Option<Entity>,      // 谁正在干
    status: JobStatus,
    created_at: u64,                 // 第几个 tick 创建的
}
```

### 6.3 分配逻辑

```
每个 tick：
1. 扫描所有未认领作业（status = Pending），按 priority 降序排序
2. 扫描所有空闲殖民者（没有 Claimed Job 的人）
3. 对每个空闲殖民者：
   a. 从作业队列中找自己能做的（技能匹配 + 距离在营地内）
   b. 选距离最近的那个
   c. 认领作业 → 殖民者有 Claimed Job → 作业标记 claimed_by
4. 认领了的殖民者：
   a. 走过去（MovementSystem per tick）
   b. 到达作业位置 → 开始执行
   c. 每 tick 推进 progress
   d. progress = 1.0 → 作业完成 → 发 JobCompleted 事件
```

营地模式下自动执行。冒险模式下正在干活的会自动回到营地继续干。

## 八、文本管线

（不改，搬之前的三层架构）

```
L1 · 即时日志
    触发事件时立即生成的描述文本
    模板格式："{actor}把{target}拖到了{place}上。"

L2 · 辐射影响
    系统自动扫描离事件中心 N 格内的角色生成的反应文本
    逻辑：角色距离 + 性格 + 当前状态 → 生成不同反应

L3 · 记忆沉淀
    拥有相关记忆的角色在合适的时机触发回忆
    逻辑：随机触发条件 + 记忆类型 → 生成回忆文本
```

## 九、存档接口

```rust
struct SaveData {
    // 世界
    seed: u64,
    chunks: HashMap<ChunkCoord, ChunkSnapshot>,
    bubble_center: (i64, i64),

    // 营地
    camp_boundary: Option<((i64, i64), (i64, i64))>,
    job_queue: Vec<JobSnapshot>,

    // 实体
    entities: Vec<EntitySnapshot>,

    // 全局
    tick_id: u64,
    reputation: i32,
    faction_states: HashMap<String, FactionState>,
}
```

每个 `ChunkSnapshot` 是区块的完整快照（瓦片+建筑+物品），只保存修改过的区块。世界生成时的区块不需要存档，重新生成结果一样（基于 seed）。

## 十、技术栈

### 规划选型（蓝图）

| 层 | 选型 | 理由 |
|----|------|------|
| 引擎/ECS | Rust + bevy_ecs | 仿真密集，SoA 内存布局，多智能体并行 |
| UI | egui (立即模式) | 文本渲染快，中文支持好 |
| 数据 | .ron + serde | 调数值不改代码，Mod 天然就绪 |
| 噪声 | noise-rs | Perlin/Simplex 用于世界生成 |
| 寻路 | 自写 A* | 瓦片静态，路径可永久缓存 |
| 存档 | serde + 自定义格式 | 数据结构先行 |

### 当前实际选型（起步实现）

| 层 | 选型 | 理由 |
|----|------|------|
| ECS | hecs | 单 crate 零依赖，编译秒过，适合小规模原型 |
| UI | ratatui + crossterm | 纯终端渲染，跨平台，中文支持良好 |
| 数据 | .ron + serde | 调数值不改代码，同上 |
| 噪声 | rand（均匀分布） | 地图 500×500 预生成，无需 Perlin |
| 寻路 | 简单曼哈顿/Chebyshev | 当前只需碰撞检测和距离判断，无 A* 需求 |
| 存档 | 未实现 | — |

## 项目的文件结构（规划蓝图）

```
血壤/
├── README.md
├── 架构设计.md              ← 本文件
├── 世界观.md
├── 设计约束.md
├── 对话纪要.md
├── 游戏介绍.md
├── plans/                   ← 开发计划
│   ├── Planmini1-日夜与大地.md
│   ├── Planmini-狼AI.md
│   ├── Planmini3-翻垃圾.md
│   ├── 04-村庄.md
│   └── completed/           ← 已完成的历史 plan
├── assets/data/             ← .ron 数据文件
│   ├── terrain.ron
│   ├── food.ron
│   └── actors.ron
└── src/                     ← 源码（当前实际实现）
    ├── main.rs              ← 入口 + 模块声明
    ├── app.rs               ← App 主循环、GameMap 属性、生成逻辑
    ├── components.rs        ← ECS 组件定义（Player/Health/Pile/Wall/Door...）
    ├── world.rs             ← GameMap 结构（500×500 + roof + window_light）
    ├── events.rs            ← GameEvent 枚举
    ├── data.rs              ← .ron 数据加载
    ├── items.rs             ← ItemKind + 物品定义
    ├── narrative.rs         ← 三层文本管线
    ├── desc.rs              ← 诙谐描述系统
    ├── village.rs           ← CDDA palette 村庄生成
    ├── systems/             ← ECS 系统（全扁平，不分目录）
    │   ├── mod.rs
    │   ├── input.rs         ← 键位分发 + F6调试 + 观察状态
    │   ├── movement.rs      ← 移动 + 碰撞
    │   ├── combat.rs        ← 狼AI + 攻击
    │   ├── examine.rs       ← e 键交互（门/床/容器/墙）
    │   ├── harvest.rs       ← 砍树/挖矿/采摘 + 砸墙
    │   ├── eat.rs           ← 进食
    │   ├── needs.rs         ← 需求衰减
    │   ├── nature.rs        ← 灌木生长
    │   ├── dark.rs          ← 刑讯
    │   ├── reaction.rs      ← 反噬反应
    │   ├── interact.rs      ← 捡取/丢弃
    │   └── ai.rs            ← AI 系统
    └── ui/                  ← ratatui 终端界面
        ├── mod.rs           ← 界面编排 + 实体 glyph + 绘制优先级
        ├── map_view.rs      ← 地图/视口渲染
        ├── side_panel.rs    ← 侧栏 Tab + 观察部件
        ├── inspect_popup.rs ← e 键物品弹窗
        ├── debug_popup.rs   ← F6 调试弹窗
        └── log_view.rs      ← 事件日志
```
