# AI 约束

> 不管哪个模型、哪个 session、哪个工具——接手这个项目就必须守这些规矩。
> 目的是让代码不堆屎山、项目结构不腐化、每次改动都留下可追溯的上下文。

---

## 一、代码修改铁律

| 规则 | 说明 |
|------|------|
| **改前必读** | 改任何文件前，先 Read 目标文件和所有被它引用的模块。凭记忆写 = 等着被打脸。 |
| **超过 3 文件先列清单** | 列出变更范围（涉及哪些文件、为什么、可能断什么），等作者点头再动手。 |
| **cargo check 必须绿** | 改完 `cargo check` 没过别交差。编译不过的代码是未完成品。 |
| **别顺手改** | 修 bug A 的时候发现"这个地方写法不太好"——记下来，别顺手改。顺手的事是屎山的主要来源。如果确实需要改，先确认。 |
| **reserve 现有风格** | 新代码跟周围代码保持一致的命名、缩进、注释密度。别在一堆 `snake_case` 里塞 `camelCase`。 |

---

## 二、架构边界

| 规则 | 说明 |
|------|------|
| **系统间只走事件总线** | `src/systems/*.rs` 之间不互相调内部函数。一个系统的输出 → `GameEvent`，其他系统消费 `GameEvent`。加新系统不改旧的。 |
| **GameMap 封装** | `tiles`、`roof`、`window_light`、`revealed` 是私有的。别给它们加 `pub`——用公开方法（`tile()`、`is_walkable()`、`has_roof()` 等）访问。 |
| **一个模块干一件事** | 单个 `.rs` 文件超过 500 行 → 考虑拆。组件定义归 `components.rs`，数据加载归 `data.rs`，别混。 |
| **ECS 纯度** | 数据在组件里，逻辑在系统里。别在组件上 impl 业务逻辑方法（`clamp()` 这种数据校验可以）。 |
| **别碰 main.rs** | `main.rs` 只负责启动：加载数据、new App、进 loop。业务逻辑挪到对应的系统文件或 app.rs 里。 |

## 三、依赖控制

| 规则 | 说明 |
|------|------|
| **加新 crate = 必须说明理由** | 每个新依赖都要在对话纪要里记录：加了什么、为什么、替代方案是什么、为什么替代方案不行。 |
| **std 优先** | 能用标准库搞定的别拉第三方。Rust 的 `HashMap`、`Vec`、`String` 够覆盖 90% 的需求。 |
| **三次原则** | 同一个模式出现第三次，才允许抽象。第一次写死，第二次复制，第三次重构——别在第一次就搞"以后可能需要"的泛型。 |
| **锁定版本** | `Cargo.toml` 里依赖写死主版本号（`ratatui = "0.29"`）。别用 `"*"` 或 `">=0.1"`。 |

## 四、测试规矩

| 规则 | 说明 |
|------|------|
| **新增功能 >50 行 → 带测试** | 新系统、新交互、新组件——超过 50 行就必须有对应的 `#[test]`。不要求覆盖率 100%，但核心路径要覆盖。 |
| **修 bug → 先复现进测试** | 修之前写一条测试复现 bug（test 先红），修完 test 变绿。防止同一個坑摔两次。 |
| **unwrap 管控** | `unwrap()` 只允许在：① `#[test]` 函数里 ② `main.rs` 启动阶段 ③ 已经逻辑上不可能失败的场景（此时注释说明为什么不可能）。其他地方全部用 `?` 或 `match`。 |
| **测试放源文件里** | 用 `#[cfg(test)] mod tests {}` 写在对应源文件底部。别急着建 `tests/` 集成测试目录——等单元测试铺到 30+ 条再说。 |
| **test_new 用于轻量构造** | `GameMap::test_new(w, h)` 创建一个全是草地的迷你地图用于测试。别在测试里跑完整的 `GameMap::generate()`——浪费时间和 CPU。 |

## 五、代码干净度

| 规则 | 说明 |
|------|------|
| **注释掉的代码直接删** | 别留着"以后可能用"——git 有历史记录。注释掉的代码在 PR 里被发现，默认视为垃圾。 |
| **TODO 带日期** | `// TODO(2026-07-10): 水源系统接入后改这里`。超过两周没动的 TODO → 要么转成 issue，要么删。 |
| **变量名别缩写到看不懂** | `spatial_idx` 可以，`si` 不行。`entity_label` 可以，`el` 不行。代码是写给人看的，编译器不在乎你省几个字符。 |
| **魔法数字必须命名** | `if dist > 15` → 错。`const FLEE_SAFE_DIST: i32 = 15; if dist > FLEE_SAFE_DIST` → 对。 |
| **无用的 `mut`/`use`/`allow` 全删** | `cargo clippy` 会帮你找。每次改完跑一遍，warning 清零。 |

## 六、Plan 执行机制

### 6.1 难度定级（开工前必须报）

收到任何 Plan/功能请求后，第一件事报难度等级，帮作者分配模型：

| 级别 | 特征 | 推荐模型 |
|------|------|----------|
| **S** — 架构级 | 新系统设计、跨 5+ 文件重构、数据模型变更、ECS 组件拆分合并 | Pro（DeepSeek V4 Pro / Opus） |
| **A** — 系统级 | 单系统实现、现有系统大改、涉及 3-5 文件 | Pro，简单部分可外包 Flash |
| **B** — 功能级 | 加新交互、扩展现有功能、2-3 文件改动 | Flash 起手，卡住切 Pro |
| **C** — 修补级 | 修 bug、改文案、调数值、单文件改动 | Flash 随便干 |

### 6.2 判定公式

```
分数 = max(涉及文件数, 新概念数) + (改ECS组件 ? +1 : 0) + (改GameMap ? +1 : 0)
S级: ≥ 6  |  A级: ≥ 4  |  B级: ≥ 2  |  C级: < 2
```

### 6.3 执行后自审

Plan 执行完后，往对话纪要记一笔：
- 实际比预估多/少改了什么文件
- 有没有产生新的技术债
- 哪一步可以做得更简单
- 有没有发现值得修的预存问题

### 6.4 禁止越界

Plan 里没写的东西**别顺手加**。想到了记下来等下次 Plan，别在实现 A 功能的时候给 B 系统重构了——这他妈是屎山最大的来源。

## 七、对话效率

| 规则 | 说明 |
|------|------|
| **三段式回答** | 判断 → 理由 → 方案。别铺垫，别叙述，别展开成论文。 |
| **问干不干时直接答** | 需要"干"或"不干，因为……"。别又分析一遍。 |
| **技术判断必须准** | 嘴臭可以，技术不能错。为了俏皮话把技术判断搞错了 → 等于 code review 被拒。 |
| **关键操作确认** | 删文件、git push --force、改数据文件——正经确认，别用嘴臭蒙混过去。 |
| **记进对话纪要** | 每次重要的技术决策、方向转变、发现的问题——写进 `docs/dev/对话纪要.md`。下次换模型/session 时，别的 AI 能接上。 |

## 八、文档体系

```
docs/
├── design/          ← 游戏设计（世界观/架构/约束/介绍）
│   ├── 游戏介绍.md
│   ├── 世界观.md
│   ├── 架构设计.md
│   └── 设计约束.md
├── dev/             ← 开发协作（对话记录/AI约束）
│   ├── 对话纪要.md
│   └── AI约束.md    ← 本文件
└── plans/           ← 开发计划（已完成即删）
```

- **保留**：当前有效的 plan、已完成的小型 Planmini
- **删除**：体量大的已完成计划（早期愿景文档），避免混淆
- **规则**：plan 完成后，把执行记录写进对话纪要，然后删 plan 文件。不养坟头。
